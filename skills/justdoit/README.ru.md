# Скилл justdoit

Скилл для Codex и Claude Code, который превращает расплывчатую задачу в понятный execution pack и спрашивает подтверждение перед стартом исполнения.

## Проблема

Большинство длинных run разваливаются ещё до кода:
- репозиторий не был нормально просмотрен
- план существует только в чате
- execution начинается до того, как определены проверки
- пользователь получает простыню промптов вместо понятного "что будет дальше"

## Решение

`justdoit` заставляет агента вести себя как execution-оператор:
- сначала изучить репозиторий
- собрать `plans.md`, `status.md` и `test-plan.md`
- коротко объяснить запуск на человеческом языке
- попросить подтверждение перед execution
- перейти в execution loop только после согласия

## Установка

### Codex

В Codex можно просто написать:

```text
Use $skill-installer to install https://github.com/serejaris/justdoit/tree/main/skills/justdoit
```

Этого достаточно, чтобы другой агент Codex сам установил скилл из репозитория.

Ручная установка:

```bash
cp -R skills/justdoit ~/.codex/skills/
```

После установки перезапусти Codex.

### Claude Code

Формат `SKILL.md` совместим со скиллами Claude Code. Скопируй директорию скилла:

```bash
# На уровне пользователя (доступно во всех проектах)
mkdir -p ~/.claude/skills
cp -R skills/justdoit ~/.claude/skills/

# На уровне проекта (только в этом проекте)
mkdir -p .claude/skills
cp -R skills/justdoit .claude/skills/
```

После установки используй `/justdoit <задача>` в Claude Code.

## Когда Использовать

`justdoit` стоит использовать по умолчанию почти для любой нетривиальной задачи.

Сюда входит:
- продуктовая задача
- задача на код
- разбор PRD в execution
- изменения в репозитории, которые лучше сначала осмыслить, а потом запускать
- любая задача, где агент должен сначала понять проект, а потом предложить run

То есть специальная фраза обычно вообще не нужна.
Если задача требует реальной работы, а не ответа в одну строчку, `justdoit` должен быть дефолтным режимом.

## Что Получаешь На Выходе

- `plans.md` как source of truth
- `status.md` как живой журнал run
- `test-plan.md` как набор проверок и acceptance gates
- короткое предложение на запуск вместо dump промптов

## Ключевое Поведение

- планирование с опорой на реальный repo context
- dependency-safe milestones
- validation-first execution
- общение на product-level
- явное подтверждение перед стартом
